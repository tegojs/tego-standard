租户隔离
1. 评估下这个系统里,我想顶层分公司管理账目,该怎么做,有什么想法吗
我先快速看一下这个项目里现有的账目/组织架构相关设计（比如租户、公司、分支机构、财务模块），再给你一个贴合当前系统的顶层方案，而不是泛泛建议。
可以做，而且你这个诉求和当前系统能力是比较契合的。基于现有架构，我建议走 “组织树 + 行级权限 + 结算单” 这条路，先做 MVP。
建议方案（贴合你现在系统）
- 组织建模：直接复用现有 departments 树，把“总公司/分公司”映射为部门层级（总公司=根节点，分公司=子节点）。
- 账目核心模型：补 3 张业务表就能跑起来：orders（业务单据）、settlementStatements（结算单）、reconciliationItems（对账差异）。
- 权限策略：复用 ACL，做 3 类角色：hq_finance（总部财务）、branch_finance（分公司财务）、auditor（审计只读）；分公司默认只看本分公司数据，总部看全量。
- 汇总报表：复用现有 charts:query 聚合能力，先出“分公司维度营收/应结/已结/差异”看板。
- 流程闭环：按“生成结算单 -> 差异处理 -> 总部确认”跑，先别上太复杂总账，后续再加 ledgerEntries 不可变流水。
为什么这样做
- 你现有系统已有：
  - 组织树（部门）能力；
  - 细粒度 ACL（资源/动作/scope）；
  - 聚合统计能力（图表查询 + workflow aggregate）。
- 真正缺的是“财务业务实体+流程”，不是底层框架能力。
最小上线版本（2-4 周可控）
- 第 1 周：建表 + 基础 CRUD + ACL 配置
- 第 2 周：结算单生成/确认 API + 差异项处理
- 第 3 周：看板与列表页（总部视角/分公司视角）
- 第 4 周：优化（重算、导出、审计日志）
关键取舍建议
- 先单库多分公司（通过 branchId 行级隔离）更快；不要一开始做多应用强隔离。
- 先“结算单”后“总账”；先把管理闭环跑通，再引入严谨会计流水。
- 先月结，再日结；降低复杂度，减少异常处理量。

2. 目前这个系统是个低代码平台. 抽象的逻辑是要在同一套数据表体系和工作流逻辑,以及页面配置的前提下, 将不同实体的数据分离开.互不干扰
你这个目标本质上是：“共享元数据（表结构/流程/页面）+ 数据按实体强隔离”，在低代码平台里是非常典型的 Tenant Isolation（租户隔离） 设计。
建议你定一个统一抽象：把“不同实体”定义为 tenant（可以叫公司、客户、业务主体都行）。
推荐总方案（单套配置，多实体隔离）
- 共享层：数据表定义、工作流定义、页面 schema、权限资源定义，全平台共用一份。
- 隔离层：所有业务数据都带 tenantId，任何读写都自动附带 tenantId 条件。
- 上下文层：请求进入时解析当前 tenantId（域名、header、登录态、切换器），注入到全链路上下文。
- 防穿透层：ACL scope + 查询拦截 + 工作流执行上下文 + 唯一索引约束，四层同时兜底。

---
你现在系统里最适合的落地方式
你已有 ACL、组织树、工作流、图表聚合能力，直接做下面几件事就够：
- 统一租户字段
  - 在所有“业务表”加 tenantId（不要漏中间表/日志表/任务表）。
- 全局查询拦截器
  - list/get/update/destroy 默认自动 where tenantId = currentTenantId。
  - 禁止前端自由覆盖 tenantId 条件（防越权）。
- 创建写入强制注入
  - create/import/workflow-create 强制写入当前 tenantId，忽略客户端传入值。
- ACL 增加固定 scope
  - 类似 scope: tenant，表达式就是当前租户过滤；平台管理员可有 all。
- 工作流租户继承
  - 触发器读到哪条记录，就把 tenantId 贯穿到后续节点（查询、写入、聚合、通知任务）。
- 图表/报表强制租户过滤
  - 聚合接口在服务端再补一层 tenantId 条件，不能只信前端配置。
- 唯一约束改为“租户内唯一”
  - 例如 (tenantId, code)，而不是全局 code，避免实体间冲突。
- 审计与排障
  - 审计日志记录 tenantId，方便排查串数据问题。

---
两种隔离级别怎么选
- 行级隔离（推荐你现在先用）
  - 优点：复用同一套 schema/流程/页面，迭代快，符合你诉求。
  - 风险：必须把拦截器和权限做严，防漏条件。
- 库级/应用级隔离（你系统已有 multi-app 能力）
  - 优点：天然更强隔离。
  - 缺点：配置、升级、运维成本高，不符合“同一套配置复用”初衷。
你当前诉求明显更匹配 行级隔离 + 强约束兜底。

---
最容易踩坑的 5 个点
- 忘记给某些“次要表”加 tenantId（附件、评论、异步任务、消息表）。
- 工作流里的“自定义查询节点”没自动带租户条件。
- 导入导出接口没做租户注入/过滤。
- 缓存 key 没带 tenantId，导致数据串读。
- 后台批处理任务没设置租户上下文，跑成全量。

3. 给一份改造清单
下面给你一份可直接排期的 低代码平台“单套配置 + 多实体数据隔离”改造清单。我按你当前系统能力（ACL、workflow、charts、department、多应用）来写，尽量可执行。
3.1 目标与边界（先定死）
- 目标：同一套表结构/流程/页面配置，按 tenantId 隔离业务数据，实体间互不干扰。
- 边界：本次先做 行级隔离（单库），不做多库拆分。
- 超级管理员例外：允许跨租户查看，但必须审计留痕。

---
3.1 数据层改造（最高优先级）
- 统一租户主键
  - 所有业务表新增 tenantId（含中间表、日志表、任务表、附件表、消息表）。
- 索引/唯一键调整
  - 高频查询加 (tenantId, createdAt)、(tenantId, status)。
  - 唯一键改为租户内唯一：(tenantId, code)。
- 历史数据回填
  - 为存量数据补 tenantId（脚本+回滚方案）。
- DDL 规范
  - 新表必须包含 tenantId，作为创建模板规则。
验收标准
- 任意业务表可通过 tenantId 快速过滤。
- 无全局唯一约束残留（除平台级系统表）。

---
3.2 请求上下文与网关注入
- 租户识别策略
  - 优先级建议：X-Tenant-Id > 登录态绑定 > 二级域名映射。
- 上下文注入
  - 在请求入口统一写入 ctx.state.currentTenant。
- 防伪造
  - 普通用户不可自带任意 tenantId；仅平台管理员可切换，且记录审计日志。
验收标准
- 每个请求都能稳定拿到 currentTenant.id。
- 非管理员传其他租户 ID 会被拒绝。

---
3.3 ORM/资源层强制隔离（关键）
- 读操作拦截
  - list/get/count/aggregate 自动追加 tenantId = currentTenant.id。
- 写操作拦截
  - create/import/batchCreate 强制注入 tenantId，忽略客户端传值。
  - update/delete 自动限制 where tenantId = currentTenant.id。
- 防绕过
  - 禁止通过原始查询绕过租户条件；提供“安全查询 API”。
验收标准
- 前端即使不传租户条件，也不会串数据。
- 恶意传别的 tenantId 不生效。

---
3.4 ACL 权限模型补强
- 新增标准 scope
  - tenant: 当前租户数据。
  - all: 平台管理员跨租户。
- 资源动作模板
  - 对业务资源统一套 tenant scope。
- 字段级防护
  - 屏蔽普通角色对 tenantId 的写权限。
- 角色体系
  - platform_admin（跨租户）、tenant_admin、tenant_user、auditor。
验收标准
- 普通角色永远只见本租户数据。
- 管理员跨租户访问有日志。

---
3.5 工作流引擎改造（最容易漏）
- 触发上下文继承
  - workflow 从触发记录继承 tenantId。
- 节点执行隔离
  - 查询/聚合/写入节点默认带 tenantId 过滤/注入。
- 定时任务
  - 定时任务按租户分片执行，不跑全量混查。
- 异常补偿
  - 补偿与重试任务保留 tenantId 上下文。
验收标准
- 任一流程实例只读写本租户数据。
- 定时任务不会跨租户处理。

---
3.6 图表与报表（聚合口防穿透）
- 服务端二次过滤
  - charts:query 类聚合接口强制拼接 tenantId 条件。
- 缓存隔离
  - 报表缓存 key 必须包含 tenantId。
- 导出隔离
  - 导出任务按租户过滤，文件命名和存储路径带租户前缀。
验收标准
- 报表总计与业务列表一致（同租户口径）。
- 缓存命中不串租户。

---
3.7 页面配置与低代码运行时
- 配置与数据分层
  - 页面/表单 schema 保持全局共享；运行态数据请求自动租户隔离。
- 变量系统
  - 内置 currentTenant 变量供表达式使用，但不允许前端改写。
- 组件封装
  - 列表、关联选择器、统计卡片等基础组件默认自动带租户条件。
验收标准
- 同一页面配置在不同租户登录下显示不同数据。
- 无需页面作者手工写租户过滤。

---
3.8 异步系统与外部集成
- 消息队列
  - topic/payload 带 tenantId。
- Webhook/API 集成
  - 出入参都带租户上下文；签名与幂等键建议包含 tenantId。
- 文件存储
  - 对象存储路径按 tenantId/ 分目录。
验收标准
- 异步链路全程可追踪租户。
- 外部回调不会写入错误租户数据。

---
3.9 审计、监控、告警
- 审计日志
  - 记录：操作者、角色、tenantId、资源、动作、条件、结果。
- 安全告警
  - 告警规则：跨租户访问尝试、无 tenant 条件查询、异常大批量导出。
- 可观测性
  - trace/log/metric 统一打 tenantId tag。
验收标准
- 可定位“谁在什么时间访问了哪个租户的数据”。
- 发生串数据风险时有实时告警。

---
3.10 测试与发布策略（必须做）
- 自动化测试矩阵
  - 单元：查询拦截、写入注入、ACL scope。
  - 集成：A/B 两租户并发读写互不影响。
  - 回归：工作流、图表、导出、定时任务。
- 压测
  - 多租户并发场景下索引命中与慢查询分析。
- 灰度发布
  - 先开新租户，再迁老租户；支持一键回退开关。
验收标准
- 核心隔离用例 100% 通过。
- 灰度阶段无跨租户事故。

---
建议排期（4 个迭代）
- 迭代1：数据层 + 请求上下文 + 资源层强制隔离。
- 迭代2：ACL scope + 页面运行时默认隔离 + 导出改造。
- 迭代3：工作流/定时任务/图表聚合全链路隔离。
- 迭代4：审计告警 + 压测 + 灰度迁移
