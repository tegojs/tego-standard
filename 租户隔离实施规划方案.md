# 租户隔离能力状态与后续计划

本文档是 `feat/platform-tenant-isolation` 分支的租户隔离权威文档。早期问答稿、阶段计划、复核笔记和临时设计文档已合并到本文档；后续以本文档记录当前状态、模块适配范围、未启用边界、测试缺口和上线前工作。

## 目标与边界

租户隔离能力面向低代码平台的「单套配置，多租户数据隔离」场景：

- 数据表定义、页面 schema、工作流定义、权限资源和插件配置尽量复用同一套元数据。
- 业务数据按租户隔离，默认由服务端强制处理，不依赖页面作者或业务插件手工追加 `tenantId` 条件。
- 用户通过当前租户上下文访问数据；平台管理员可以受控代入目标租户，并保留审计信息。
- 本分支采用行级隔离，不做多数据库、多部署实例或多应用强隔离。
- 首期主体能力落在标准模块仓库内，不以前置修改 `@tego/server` 核心为条件。
- 历史业务数据归类迁移工具不纳入本分支范围；旧数据可见性通过兼容策略控制。

## 核心模型

平台统一使用以下概念：

| 概念 | 说明 |
| --- | --- |
| `tenant` | 租户实体，表示数据隔离边界。 |
| `tenantId` | 租户化业务记录的归属字段。 |
| `currentTenant` | 当前请求、页面或后台任务使用的租户上下文。 |
| `tenantScoped` | 仅当前租户可见的数据。 |
| `tenantInherited` | 当前租户及其子孙租户可见的数据。 |
| `shared` | 平台共享数据，不参与租户隔离。 |
| `legacyDataTenantIds` | 旧数据读可见配置，只影响读取，不放宽更新或删除。 |

集合租户语义统一通过 `collections.options.tenancy` 表达。`tenantScoped` 与 `tenantInherited` 集合使用 `tenantId` context field：

```ts
{
  type: 'context',
  name: 'tenantId',
  dataIndex: 'state.currentTenant.id',
  createOnly: true,
}
```

请求上下文标准字段：

- `X-Tenant-Id`
- `ctx.state.currentTenant`
- `ctx.state.currentTenantId`
- `ctx.state.currentTenantDescendantIds`
- `ctx.state.currentTenancyMode`
- `ctx.state.currentLegacyDataTenantIds`

## 重要架构决策

### 不把租户隔离迁入 `module-data-source`

`module-data-source` 负责数据源连接、加载、ACL 同步和数据源生命周期管理；租户隔离是业务策略，仍由 `module-tenant` 拥有。`tenantScoped`、`tenantInherited`、`legacyDataTenantIds`、租户树继承等语义不应下沉到数据源基础设施层。

其他插件和模块的适配原则：

- 运行时代码不能静态依赖 `@tachybase/module-tenant`，因为租户模块可能未安装或未启用。
- 其他模块通过 `ctx.state.currentTenant`、租户上下文和本地 no-op helper 做适配；无租户上下文时自然退化。
- 测试代码可以在 workspace 内引用 `module-tenant`，用于防漂移验证；运行时代码不可以。

### SQL/View/Workflow SQL 不自动隔离

以下路径不由框架自动注入租户过滤：

- SQL 集合执行（`sqlCollection:*`）
- View 集合查询（`dbViews:*`）
- Workflow SQL 指令

这些能力允许原始 SQL 或数据库视图，框架无法可靠解析并重写所有 SQL。上线前必须通过权限和配置治理控制风险：普通租户用户默认不应拥有配置或执行权限，仅 root、平台管理员或明确授权角色可用。

## 当前已完成能力

### 租户模块

- 已新增 `packages/module-tenant` 前后端入口。
- 已支持 `tenants`、`tenantUsers`、用户默认租户、当前租户解析和租户切换接口。
- 已支持租户树：`parentId`、`parent`、`children`、`path`、子孙租户查询、移动校验、循环引用保护、删除非叶子节点保护、`path` 长度友好校验。

- `setCurrentTenant` 已支持按 `X-Tenant-Id`、默认租户和可访问租户解析当前上下文。

## 租户模块未启用时的适配边界

当前复核结论如下：

- `module-user` 只在 `this.app.pm.get('tenant')?.enabled` 且集合声明 `tenantScoped/tenantInherited` 时自动注入 `tenantId`。
- `module-workflow` 使用本地 `helpers/tenant-context.ts`，运行时代码不静态导入 `@tachybase/module-tenant`；没有租户上下文时 no-op。
- `module-workflow` 与 `module-file` 的 tenant 依赖只用于开发和测试边界，不是运行时强加载路径。
- 前端租户菜单、租户管理页、集合模板中的 `tenancy/legacyDataTenantIds` 配置由 `module-tenant` 客户端插件注册；tenant 客户端插件未加载时不会暴露这些配置项。
- `APIClient` 发送 `X-Tenant-Id` 只是请求头行为；tenant 服务端模块未启用时没有对应消费逻辑。

## 合并前结论与剩余关注项

截至最终复核，完整测试、服务端租户专项、客户端租户专项和关键插件适配测试已通过；未发现阻塞合并的代码缺陷。以下内容属于上线前环境确认或上线后持续观察项，不再作为本分支代码合并阻塞项。

### 1. SQL/View/Workflow SQL 权限治理

SQL 集合、数据库视图和 Workflow SQL 是明确设计边界，框架不会自动注入租户过滤。上线前需要在生产角色配置中确认：

- 普通租户用户不能访问 `sqlCollection:*`。
- 普通租户用户不能访问 `dbViews:*`。
- 普通租户用户不能配置或执行 Workflow SQL 指令。
- root、平台管理员和显式授权角色的权限边界符合预期。

上线后重点观察这些能力的使用审计，避免被误配置给普通租户角色。

### 2. 租户模块未启用回归

未启用 `module-tenant` 时的回归覆盖已补齐，当前结论如下：

- 头像菜单不显示租户切换入口。
- 租户管理页面、租户相关 API 和租户操作不会暴露给普通应用。
- 图表、导入、导出、文件、工作流、审计等适配代码不因缺少租户上下文而报错。
- 本地 helper 保持 no-op。
- 裸 `X-Tenant-Id` header 不会在租户模块未启用时被消费为租户上下文。

### 3. 外部数据源链路

非主数据源依赖 `dataSourceManager` 补偿链，已覆盖主要中间件顺序和过滤边界。上线环境仍建议用真实外部数据源做一次冒烟：

- 已存在外部数据源的资源请求。
- 新增外部数据源后的资源请求。
- 外部数据源 reload 后的资源请求。
- `auth`、`acl`、`dataSource`、租户上下文解析的中间件顺序。

### 4. 本地 helper 防漂移测试

图表、导出、工作流各自保留了本地租户 helper，这是为了避免运行时硬依赖 `module-tenant`。当前已通过防漂移测试确认：

- `plugin-block-charts` helper 与 `module-tenant` 权威行为一致。
- `plugin-action-export` worker helper 与 `module-tenant` 权威行为一致。
- `module-workflow` tenant context helper 与 `module-tenant` 权威行为一致。

### 5. 运维闭环

安全事件已经写入审计日志，灰度期间重点观察：

- `tenant_cross_tenant_attempt`：普通用户尝试访问不可访问租户。
- `tenant_impersonation`：root 或平台管理员代入目标租户。
- `tenant_bulk_export_alert`：单次大批量导出。
- `tenant_access_denied`：请求缺少有效租户上下文或租户未启用。

### 8. 灰度和回退 ✅ 已形成方案

灰度启用范围、启用前检查、启用后观测、回退策略和历史数据归类方案已补充到下方「灰度和回退」小节（8.1–8.5）。

#### 8.1 灰度启用范围

分三阶段启用 `module-tenant`，每阶段观察至少一个完整业务周期（建议 3–5 个工作日）：

| 阶段 | 范围 | 目的 |
| --- | --- | --- |
| P0 内部验证 | 仅平台管理员 + 1 个内部测试租户 | 验证中间件链路、迁移、审计事件、附件/导出路径 |
| P1 小规模灰度 | 2–3 个非核心业务租户（建议选数据量小、工作流少的租户） | 验证真实业务 CRUD、导入导出、工作流触发、图表查询 |
| P2 全量放开 | 所有租户 | 正式上线 |

每阶段启用前，先在 `tenants` 表将目标租户 `enabled` 设为 `true`，并通过 `X-Tenant-Id` header 切换到目标租户执行冒烟验证。

#### 8.2 启用前检查清单

**迁移验证**

```sql
-- 确认三张表的租户字段已存在
SELECT column_name FROM information_schema.columns WHERE table_name = 'executions' AND column_name = 'tenantId';
SELECT column_name FROM information_schema.columns WHERE table_name = 'auditLogs' AND column_name = 'tenantId';
SELECT column_name FROM information_schema.columns WHERE table_name = 'attachments' AND column_name = 'tenantId';
```

- 迁移脚本幂等，重复执行不会报错；但首次启用前必须确认迁移已完成。
- 迁移仅添加 `allowNull: true` 的新列和索引，不做 `ALTER DROP`，对已有数据无破坏性。

**权限确认**

- 普通租户用户不能访问 `sqlCollection:execute`（需 `pm.database-connections.collections` snippet）。
- 普通租户用户不能访问 `dbViews:query`（受 ACL 默认拒绝策略保护）。
- 普通租户用户不能创建/编辑 Workflow SQL 节点（需 `pm.workflow.sql` snippet）。
- root、平台管理员和显式授权角色的边界符合预期。

**附件路径**

- 确认 `storage/uploads/tenants/<tenantId>/` 目录可写。
- 确认对象存储（如 S3/MinIO）的 key 前缀策略兼容 `tenants/<tenantId>/` 子路径。
- 确认历史附件（无 `tenantId`）仍可通过原路径访问。

**导出路径**

- 确认异步导出文件保存路径 `{root}/tenants/<tenantId>/` 可写。
- 确认导出 URL 路径 `{basePath}/tenants/<tenantId>/<fileName>` 可被前端正确下载。
- 确认 `BULK_EXPORT_THRESHOLD`（默认 1000 行）阈值符合业务预期。

**审计事件**

- 确认 `plugin-audit-logs` 已启用且 `auditLogs` 表迁移完成。
- 确认 `tenant.securityViolation` 事件监听器已注册（有重复注册保护）。

#### 8.3 启用后观测

灰度期间重点监控 `auditLogs` 表中的四类安全事件：

| 事件类型 | 排查口径 | 处置建议 |
| --- | --- | --- |
| `tenant_access_denied` | 查询 `SELECT * FROM "auditLogs" WHERE type = 'tenant_access_denied' ORDER BY "createdAt" DESC LIMIT 50`。关注 `collectionName` 和 `userId`。 | 通常是请求缺少 `X-Tenant-Id` header 或租户未启用。检查客户端 APIClient 配置和租户 `enabled` 状态。 |
| `tenant_cross_tenant_attempt` | 查询 `SELECT * FROM "auditLogs" WHERE type = 'tenant_cross_tenant_attempt' ORDER BY "createdAt" DESC LIMIT 50`。关注 `userId` 和 `details`。 | 用户尝试访问非其归属租户的数据。确认 `tenantUsers` 关联是否正确，或是否存在恶意越权。 |
| `tenant_impersonation` | 查询 `SELECT * FROM "auditLogs" WHERE type = 'tenant_impersonation' ORDER BY "createdAt" DESC LIMIT 50`。关注 `actorUserId` 和 `impersonatedTenantId`。 | root 用户代入其他租户。属于正常运维行为，但需定期审计确认合理性。 |
| `tenant_bulk_export_alert` | 查询 `SELECT * FROM "auditLogs" WHERE type = 'tenant_bulk_export_alert' ORDER BY "createdAt" DESC LIMIT 50`。关注 `collectionName` 和 `details.rowCount`。 | 单次导出 >= 1000 行。确认是否为正常业务需求；如频繁触发，考虑调整阈值或增加审批流程。 |

**日常巡检 SQL**

```sql
-- 灰度期间每日执行：按类型统计安全事件
SELECT type, COUNT(*) AS cnt
FROM "auditLogs"
WHERE type IN ('tenant_access_denied', 'tenant_cross_tenant_attempt', 'tenant_impersonation', 'tenant_bulk_export_alert')
  AND "createdAt" > NOW() - INTERVAL '24 hours'
GROUP BY type;
```

#### 8.4 回退策略

**回退操作：禁用 `module-tenant`**

在插件管理中禁用 `module-tenant`（或从 `application.json` 移除插件配置），重启应用。

**关闭后 tenantId 字段保留**

- 三张表（`executions`、`auditLogs`、`attachments`）的 `tenantId` 字段及索引**保留不删除**。
- 迁移脚本仅做 `addColumn`，不做 `removeColumn`；禁用模块不会触发列删除。
- `ensureTenantIdField` 钩子不再注册，新集合不会自动获得 `tenantId` 字段，但已有字段不受影响。
- 后续重新启用时，字段和索引已存在，迁移幂等跳过。

**已写入租户数据的可读性验证**

- 禁用后，`tenantResourceGuard` 和 `setCurrentTenant` 中间件不再注册，请求不再注入租户过滤。
- 所有 `tenantScoped` / `tenantInherited` 集合的 CRUD 退化为无过滤全表操作，已写入的带 `tenantId` 记录**全部可见**。
- 验证方法：禁用后对关键集合执行 `list`，确认返回记录数 >= 启用前记录数（含灰度期间新增数据）。

**各子系统回退检查**

| 子系统 | 回退后行为 | 验证方法 |
| --- | --- | --- |
| 导入 | `plugin-action-import` 不再注入 `tenantId`，导入记录的 `tenantId` 为 `null` 或文件中的值 | 执行一次导入，确认记录可正常创建且 `tenantId` 不被强制覆盖 |
| 导出 | 同步/异步导出不再应用租户过滤，导出全量数据；文件路径不再包含 `tenants/<tenantId>/` | 执行一次导出，确认返回全量数据且文件路径为原始路径 |
| 附件 | 上传路径退化为 `storage/uploads/`（无 `tenants/<tenantId>/` 子路径） | 上传一个附件，确认存储路径不含 `tenants/` 前缀；历史附件仍可通过原路径访问 |
| 审计 | `auditLogs` 不再写入租户安全事件；普通 CRUD 审计仍正常工作，但 `tenantId` 等字段为 `null` | 执行一次 CRUD 操作，确认审计日志正常生成且不含租户安全事件 |
| 工作流 | `executions` 不再写入 `tenantId` 和 `tenantContext`；非 SQL 指令不再注入租户过滤 | 触发一次工作流，确认执行记录正常且 `tenantId` 为 `null` |
| 图表 | 查询和缓存不再包含租户 scope | 查看一个图表，确认数据为全量且缓存 key 不含租户标识 |

**回退后不丢数据的原因**

- 所有迁移字段 `allowNull: true`，禁用后写入的记录 `tenantId` 为 `null`，不影响插入。
- 禁用后无过滤，所有记录（含带 `tenantId` 和 `null` 的）均可读。
- 附件物理文件不因模块禁用而删除；仅新上传文件路径规则改变。

#### 8.5 历史数据归类

历史业务数据归类迁移工具**不纳入本分支范围**（与"目标与边界"一致）。旧数据（`tenantId` 为 `null`）的可见性通过 `legacyDataTenantIds` 配置控制：

- 配置了 `legacyDataTenantIds` 的集合，读操作允许 `tenantId: null` 的记录被指定租户读取。
- 写操作（`update`/`destroy`）不受 `legacyDataTenantIds` 影响，仍严格限制为当前租户数据。
- 上线后如需将旧数据归类到具体租户，由业务侧通过导入/脚本单独处理，不在框架层面提供工具。

## 重要实现原则

- 客户端传入的 `tenantId` 默认不可信。
- `legacyDataTenantIds` 只放宽读取，不放宽更新和删除。
- 平台管理员代入只改变 effective tenant，不覆盖 `currentUser`。
- 页面 schema 和插件配置尽量共享；运行时数据访问由服务端按当前租户自动筛分。
- 共享字典、平台配置、SQL/View 集合等不应被误加租户字段或租户过滤。
